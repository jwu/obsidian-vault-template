description: 根据指定日期或当天的 Git 提交内容补写对应日记
argument-hint: "[date]"
---

请根据指定日期或当天的 Git 提交内容，详细了解每个提交涉及的文件改动，并补写或更新该日期的日记。

## 1. 准备

- 如果 `$1` 不为空，目标日期为 `$1`
- 如果 `$1` 为空，目标日期为今天，使用当前系统日期，格式为 `YYYY-MM-DD`
- 先阅读 `99-system/dairy-editing-guide.md`，遵守日记编辑规则
- 目标日记文件：`01-daily-日记/<目标日期>.md`

在后续命令和文件路径中，都用解析后的 `<目标日期>`，不要继续使用空的 `$1`。

## 2. 读取当天提交

执行并阅读当天所有提交，按时间顺序处理：

```bash
git log --since="<目标日期> 00:00" --until="<目标日期> 23:59:59" --date=iso --pretty=format:'%h %ad %s' --reverse
```

如果没有提交，直接告诉我当天没有 Git 提交可用于生成日记。

## 3. 详细理解每个提交

对当天每个提交，至少查看：

```bash
git show --stat --name-status --format=fuller --find-renames <commit>
git show --find-renames --format='===== %h %s' --patch --stat <commit>
```

重点理解：
- 新增、修改、删除、重命名了哪些文件
- 每个文件的实际内容变化，而不只看 commit message
- 这些改动分别属于哪些事务：项目推进、档案归档、日记更新、调研、配置、私人数据等
- 哪些改动代表完成了待办、形成了判断、产生了后续方向
- 注意区分提交日期和实际发生日期：如果凌晨提交明显是在收尾前一天的事情，优先把事务写到实际发生的日期；当天日记只简要记录“提交收尾”，不要重复展开

## 4. 书写日记

读取 `01-daily-日记/<目标日期>.md`：
- 如果文件已存在，用 `edit` 更新日记内容
- 如果文件不存在，先检查是否已有同名或相近日期日记；确认不存在后，用 `obsidian vault=work create name="<目标日期>" path="01-daily-日记"` 创建，再用 `edit` 写入内容

书写时遵守 `99-system/dairy-editing-guide.md`，不要在 prompt 里重复展开日记格式规则。

额外要求：
- 日记要简洁，不要写成提交详解或流水账
- 不要只罗列提交；要把提交内容合并成少数几个主事务
- 每个主事务只写真实做了什么、为什么重要、形成了什么判断；避免展开过多实现细节
- 同类事项合并写，例如同一组技术档案、同一批项目笔记、同一条调研线，不要逐文件长篇解释
- 如果某个事务已经在前一天日记中展开，当天只写提交收尾或链接前一天，不要重复记录
- 优先使用已有笔记的 wikilink
- 不要改写 `## 待办`，除非提交内容明确显示待办状态变化

## 5. 完成后汇报

完成后用简洁中文告诉我：
- 更新了哪个日记文件
- 覆盖了哪些主要事务
- 如果发现提交中有不适合写进日记或需要我确认的内容，也一并说明
