审查当前 todo 体系的进度，对照近期实际工作，更新 `10-todo-待办/TODO.md` 的总览分类、当前判断和近阶段主轴。

## 1. 确定时间范围

- 如果 `$1` 不为空，以 `$1` 为起始日期（如 `2026-03-15`），截止到今天
- 如果 `$1` 为空，默认回溯最近两个月

## 2. 读取现状

- `10-todo-待办/TODO.md` — 了解当前主轴、判断和总览结构
- `10-todo-待办/` 下所有 `.md` 笔记（除 TODO.md 本身）— 了解每个专题的内容量和进展程度
- `01-daily-日记/` 下时间范围内的日记 — 了解实际做了什么

## 3. 对比分析

将 TODO.md 中的"近阶段建议主轴"与日记中的实际工作进行对比：

- 哪些主轴有实质推进？具体体现在哪些 todo 笔记的内容增长？
- 哪些主轴几乎没有进展？
- 是否出现了新方向，在日记中反复出现但 TODO.md 没覆盖？
- 是否有实际已产出的成果（编码规范、项目文档、agent skill 等），但没有对应 todo？

对每个 todo 笔记判断进度阶段：
- 未启动（几乎空壳或只有一两条链接）
- 收集中（资料持续补充但未结构化）
- 消化中（已按子主题分组、写了推荐判断或对比分析）
- 已产出（形成了独立的知识笔记或项目文档）

## 4. 更新 TODO.md

用 `edit` 更新 `10-todo-待办/TODO.md`，重点更新三个部分：

### 总览

将 `10-todo-待办/` 下所有 todo 笔记完整分类列出，以分类为主。大类划分建议：

- AI / Agent / CLI 工作流
- 游戏开发技术
- 游戏引擎技术
- 游戏创作
- 游戏编辑器开发
- 字体与渲染
- 游戏解包
- 3D 相关
- 审美 / 文化输入

每个大类末尾加一句整体进度判断（用 `>` 引用块），简洁说明哪些在推进、哪些未启动。

不在总览中列入无对应 todo 的实际产出。

### 当前判断

基于对比分析结果，重写判断段落。要点：
- 哪些线进展最大、哪些偏离了预期
- Harness Engineering 等新兴方向是否需要正式纳入主轴
- 哪些线处于"资料已收集但待消化"的瓶颈
- 哪些线短期内客观上没空推进，可以减轻压力

然后给出接下来的做法建议（2-3 条）。

### 近阶段建议主轴

给出 3 条当前最值得优先推进的方向，每条附简要说明。

## 5. 汇报

完成后用简洁中文汇报：
- 覆盖了哪个时间范围
- 主要发现（2-3 条）
- TODO.md 更新了哪些部分
