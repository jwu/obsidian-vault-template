# Application Archiving

为本项目中的应用（软件、工具、服务、CLI 等）建立或完善档案，输出到 `80-archive-档案库/Application-应用/`，并尽量保持与现有应用笔记一致的结构、语气与信息密度。

## When to use

在以下情况使用本技能：
- 用户要求为某个应用建立档案
- 用户提供一个应用名、GitHub 仓库链接、官网链接，希望整理成标准应用笔记
- 用户要求补充某个应用的功能、技术架构或同类对比
- 用户要求对某个应用按 `1~7` 制打分

## References

开始前优先读取这些参考材料：
- `../../../99-system/templates/Application Template.md`

如需对齐现有风格，可查看 `80-archive-档案库/Application-应用/` 下已有笔记，例如：
- `../../../80-archive-档案库/Application-应用/F3D.md`
- `../../../80-archive-档案库/Application-应用/aseprite.md`
- `../../../80-archive-档案库/Application-应用/cal.diy.md`

评分规则参考：
- `../../../99-system/rating.md`（`1~7` 整数制）

## Output

应用档案统一放在：
- `80-archive-档案库/Application-应用/`

默认要求：
- 文件名使用应用常用名
- 笔记需包含 `created:` 属性
- 优先沿用 `[[Application Template]]` 的结构
- 语言 wikilink 一律小写，创建前需检查 `80-archive-档案库/Technique-技术/` 中是否已有对应语言笔记

## Rules

- 默认使用简洁中文
- 保留事实，不臆测
- 优先小步补充，不大改原文
- 如果信息不确定，宁可留空，也不要强填
- 编辑 Markdown 内容时保持 Obsidian 兼容
- 引用文件夹时不要使用 `[[wikilink]]`，统一使用 `文件夹名/` 形式
- 若需创建、移动、重命名应用笔记，遵循项目要求，使用 `obsidian vault=work ...` 完成
- 应用评分统一使用 `1~7` 整数制，依据 `99-system/rating.md`
- 笔记内容中提及的技术、人物、概念等，若库内已有对应笔记则使用 wikilink，若是外部资源则使用外部链接，尽量让内容可导航

## Workflow

### 1. 确认输入

用户可能提供：
- 一个应用名（最常见，如「blender」「F3D」「cal.diy」）
- 应用名 + GitHub 仓库链接
- 应用名 + 官网或文档链接
- 应用名 + 领域描述（如「那个开源的预约调度工具」）

目标是先确认「要归档的是哪个应用」。

- 如果用户提供的信息已经足够唯一定位到该应用，则继续下一步
- 如果信息过少、存在重名，或无法确认是否为目标应用，则先向用户确认
- 在应用身份未确认前，不要擅自创建笔记或合并到现有笔记

### 2. 确认是否已有笔记

先搜索 `80-archive-档案库/Application-应用/` 中是否已有同名或近似应用笔记。

- 如果已有：在原笔记上补充
- 如果没有：基于模板新建
- 如果出现同名或疑似重名应用：先暂停并向用户确认，不要擅自合并

### 3. 信息收集与内容整理

#### 基本信息

优先从以下来源获取：
- 官网 / 文档站
- GitHub / GitLab 仓库（README、Wiki、Releases）
- 包管理器页面（npm、crates.io、pypi 等）

获取基础信息时优先使用 `uv run scrapling` 抓取网页内容，GitHub 仓库信息优先使用 `gh` 命令。

#### 封面图

部分应用有代表性的截图或封面图。获取方式：
- 官网 hero 图或 README 中的截图
- GitHub 仓库 README 中的展示图
- 如无合适图片，可省略
- 图片统一使用 `![描述](URL)` 或 `![描述|500](URL)` 格式，放在正文开头

#### 功能特性

- 从 README 的 Features 章节提取
- 从官网首页的功能列表提取
- 每个特性一行无序列表，简洁描述

#### 技术架构

- 从 README 的 Built With / Tech Stack 章节提取
- 从源码目录结构推断
- 列出核心技术栈，每项一行

#### 同类对比

在「其他」栏目中补充：
- **同类工具**：竞争或替代方案，若内部已有档案则用 wikilink
- **使用场景**：适合/不适合的场景
- **许可证**：开源许可证类型

### 4. 生产档案文件

#### 建立 frontmatter 属性

优先整理这些属性：

```yaml
---
created: YYYY-MM-DD
template: "[[Application Template]]"
website:
repository:
desc:
author:
languages:
rating:
aliases:
---
```

填写规则：
- `desc`：一句话说明这个应用是什么、做什么
- `website`：官网或主页
- `repository`：主仓库链接（GitHub / GitLab 等）
- `languages`：实现语言，以 wikilink 数组形式，如 `["[[typescript]]", "[[cpp]]"]`
    - 先在 `80-archive-档案库/Technique-技术/` 中搜索语言笔记，确保 wikilink 指向实际存在的文件名
    - 注意语言可能有多种写法：如 C++ 对应 `cpp`、`c++`、`c plus plus` 等，搜索时需覆盖常见变体
- `author`：核心作者或维护组织
- `rating`：按 `1~7` 整数制评分；未充分了解时可暂空
- `aliases`：仅在确有常见别名、缩写、旧名时填写
- 未找到或无法确认的字段，直接省略，不要保留空字段
- 仅保留已确认且对后续检索、补充有价值的属性

#### 使用统一格式写入正文

```md
## 简介

介绍一下这个应用。

## 特点

- 功能特性
- 涵盖核心功能、技术栈、优劣势
- 每个特点一行

## 技术架构

- 核心依赖、技术栈
- 每项一行

## 其他

- 同类工具（内部已有的用 wikilink）
- 适用场景
- 许可证
- 社区链接
```

格式要求：
- 简介：简要介绍这个应用是什么、解决什么问题
- 特点：用无序列表，每个特点一行；涵盖功能特性、技术栈、优劣势
- 技术架构：用无序列表，列出核心技术栈和关键依赖
- 其他：使用无序列表，注意不要重复属性中已有的信息（如仓库链接、官网）
- 如某栏目暂无可靠信息，可保留空节或不写入该节

#### 补全正文中的链接

正文写完后，必须逐项扫描「特点」「技术架构」「其他」中提到的所有技术、框架、库、工具、协议、语言、概念名称，逐个判断：

1. 在 `80-archive-档案库/` 中搜索是否已有对应笔记 → 有则用 `[[wikilink]]`
2. 没有笔记但存在明确的官网或仓库 → 用 Markdown 外部链接 `[名称](URL)`
3. 上述两项都没有 → 保留纯文本

此步骤是强制检查项，不可跳过。

## Ambiguity handling

遇到以下情况时，优先保守处理：
- 无法确认是否为同一应用：先向用户确认
- 多个来源信息冲突：优先采用官网或仓库 README；若仍无法判断，则暂不写入
- 应用版本差异大：优先记录当前稳定版本的信息，必要时标注版本
- 某栏目暂无可靠信息：可保留空节或不写入该节
- 评分不确定：宁可不评分，也不要随意打分

## Quality bar

- 优先记录「长期有价值」的功能特征与技术栈，而不是堆链接
- 简介应让读者快速判断「这个应用是否适合我的场景」
- 特点应突出差异化功能，而非面面俱到的功能罗列
- 其他中的同类对比应有实际差异点，而不是泛泛的「类似 XX」
- 如果条目很多，先选最能代表此应用的核心信息
- 若发现已有内容结构混乱，先轻量整理，再补充信息
- 输出应便于后续持续补充，而不是一次性塞满

## Delivery

完成后，简要说明：
- 新建了还是更新了哪一篇应用笔记
- 补充了哪些信息，如官网、仓库、功能特性、技术架构、同类对比
- 哪些信息仍缺失或待确认
- 若已评分，简要说明评分依据
