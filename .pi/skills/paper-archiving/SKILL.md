# Paper Archiving

为本项目中的论文建立或完善档案，输出到 `80-archive-档案库/Paper-论文/`，并尽量保持与现有论文笔记一致的结构、语气与信息密度。

## When to use

在以下情况使用本技能：
- 用户要求为某篇论文建立档案
- 用户提供一个论文链接（arXiv、OpenReview、会议页面等）或论文标题，希望整理成标准论文笔记
- 用户要求补充某篇论文的核心问题、解决思路、关键创新、还原步骤
- 用户要求对某篇论文按 `1~7` 制打分

## References

开始前优先读取这些参考材料：
- `../../../99-system/templates/Paper Template.md`

如需对齐现有风格，可查看 `80-archive-档案库/Paper-论文/` 下已有笔记，例如：
- `../../../80-archive-档案库/Paper-论文/(SIGGRAPH 2023) Learning Physically Simulated Tennis Skills from Broadcast Videos.md`
- `../../../80-archive-档案库/Paper-论文/(arXiv 2026) CuRast - Cuda-Based Software Rasterization for Billions of Triangles.md`

评分规则参考：
- `../../../99-system/rating.md`（`1~7` 整数制）

## Output

论文档案统一放在：
- `80-archive-档案库/Paper-论文/`

默认要求：
- 文件名格式为 `(venue year) Title.md`，如 `(SIGGRAPH 2023) Title.md`、`(arXiv 2026) Title.md`
    - venue 需标准化：CVPR、ICCV、ECCV、NeurIPS、ICML、ICLR、SIGGRAPH、TOG、CHI、arXiv 等
    - 若论文尚未被接收，venue 用 `arXiv`
    - 文件名中的 Title 如果过长可适当截断，保留核心信息
- 笔记需包含 `created:` 属性
- 优先沿用 `[[Paper Template]]` 的结构

## Rules

- 默认使用简洁中文
- 保留事实，不臆测
- 优先小步补充，不大改原文
- 如果信息不确定，宁可留空，也不要强填
- 编辑 Markdown 内容时保持 Obsidian 兼容
- 引用文件夹时不要使用 `[[wikilink]]`，统一使用 `文件夹名/` 形式
- 若需创建、移动、重命名论文笔记，遵循项目要求，使用 `obsidian vault=work ...` 完成
- 论文评分统一使用 `1~7` 整数制，依据 `99-system/rating.md`
- 笔记内容中提及的技术、人物、概念、相关论文等，若库内已有对应笔记则使用 wikilink，若是外部资源则使用外部链接，尽量让内容可导航

## Workflow

### 1. 确认输入

用户可能提供：
- 一个 arXiv 链接（如 `https://arxiv.org/abs/2605.15187`）
- 一个会议论文链接（如 CVPR、SIGGRAPH 等会议页面）
- 论文标题 + 会议/年份
- 论文标题 + GitHub 仓库链接
- 一个 PDF 链接

目标是先确认"要归档的是哪篇论文"。

- 如果用户提供的信息已经足够唯一定位到该论文，则继续下一步
- 如果信息过少、存在重名，或无法确认是否为目标论文，则先向用户确认
- 在论文身份未确认前，不要擅自创建笔记或合并到现有笔记

### 2. 确认是否已有笔记

先搜索 `80-archive-档案库/Paper-论文/` 中是否已有同名或近似论文笔记。

- 如果已有：在原笔记上补充
- 如果没有：基于模板新建
- 如果出现同名或疑似重名论文：先暂停并向用户确认，不要擅自合并

### 3. 信息收集与内容整理

#### 基本信息

优先从以下来源获取：
- arXiv 页面（`https://arxiv.org/abs/<id>`）
- arXiv HTML 版本（`https://arxiv.org/html/<id>v1`，包含完整论文文本）
- 论文项目主页
- GitHub / GitLab 仓库（README、代码结构）
- 会议官方页面

获取基础信息时：
- 普通网页优先使用 `uv run scrapling` 抓取内容
- arXiv 页面使用 scrapling 的 `Fetcher`（静态页面）
- GitHub 仓库信息优先使用 `gh` 命令
- YouTube 等视频信息优先使用 `yt-dlp`

#### 摘要与核心内容

从论文中提取以下结构化信息：
- **一句话摘要**：论文要解决什么问题、用什么方法、取得了什么效果
- **核心问题与挑战**：传统方法怎么做、有哪些难点；新方案面临什么挑战
- **解决思路**：核心方法是什么、具体步骤、为什么这个思路有效
- **关键创新**：与已有工作的本质区别、技术亮点、核心原理

#### 还原方法与复现

根据论文和仓库信息，整理还原/复现路径：
- 需要什么设备、环境、资源
- 还原成本（低/中/高），基于是否开源、是否需要特殊硬件、训练资源等判断
- 具体还原步骤：
    - 如果代码已开源：按代码仓库的 README / setup 流程写详细步骤
    - 如果代码未开源：根据论文方法描述给出大致方向，标注"代码未开源"
- 标注当前可还原的程度（代码级 / 方法级 / 结果级）

#### 相关论文与引用

在"核心论文引用"栏目中列出：
- 论文中引用的关键相关工作
- 每条格式：`**简称** (作者, 年份 场合) — 一句话说明`
- 优先列出最有影响力的、与本文方法最相关的引用
- 若库内已有对应的论文笔记，使用 wikilink

### 4. 生产档案文件

#### 建立 frontmatter 属性

优先整理这些属性：

```yaml
---
created: YYYY-MM-DD
template: "[[Paper Template]]"
author:
release date:
website:
paper:
video:
code:
rating:
aliases:
---
```

填写规则：
- `author`：作者列表，以 YAML 数组形式
    - 若作者较多（>5），可列出核心作者（标注 project lead / core contribution）
    - 若库内已有对应人物笔记，使用 wikilink 如 `"- [[Xue Bin Peng]]"`
- `release date`：论文发布/投稿日期（YYYY-MM-DD）
- `website`：项目主页
- `paper`：论文链接（arXiv / 会议页面 / PDF）
- `video`：如有视频演示，填入链接
- `code`：如有开源代码，填入仓库链接
- `rating`：按 `1~7` 整数制评分；未充分了解时可暂空
- `aliases`：仅在确有常见别名、缩写时填写
- 未找到或无法确认的字段，直接省略，不要保留空字段
- 仅保留已确认且对后续检索、补充有价值的属性

#### 使用统一格式写入正文

```md
![论文截图](图片链接)

一句话摘要。

## 核心问题与挑战
- 传统方法要怎么做，有哪些难点
- 新方案的挑战是什么

## 解决思路
- 解决方法
- 具体思路
- 详细步骤

## 关键创新
- 创新点是什么
- 有什么技术亮点
- 核心技术的原理

## 还原方法
- 需要什么设备环境
- 还原成本（低/中/高）
- 还原步骤概览

## 其他
- 实验亮点与关键数据
- 不足与改进空间
- 与其他工作的关系（若与库内技术/论文有关联，使用 wikilink）

## 核心论文引用
- **简称** (作者, 年份 场合) — 一句话说明

---

## 详细还原步骤

### Step 0：还原目标与可行性
说明代码是否开源、可还原程度（代码级/方法级/结果级）、依赖的特殊资源。

### Step 1：环境准备
具体命令与配置。

### Step N：最容易出问题的地方
列出常见坑点和注意事项。
```

格式要求：
- 每个节使用二级标题 `##`
- 正文无序列表用 `-` 开头，缩进 4 空格表示子级
- 代码块标注语言（bash、python 等）
- 详细还原步骤按 Step 0、1、2... 编号，每步有明确的标题和内容
- 若论文代码未开源，"详细还原步骤"可大幅简化，重点说明方法描述和还原方向

#### 论文截图选择

论文笔记开头必须放一张代表性截图（`![论文截图](图片链接)`），选择优先级如下：

1. **arXiv HTML 版本中的 Figure 1 / teaser 图**
    - arXiv HTML 页面（`https://arxiv.org/html/<id>v1`）中的 `x1.png` 通常是论文 teaser
    - URL 格式为 `https://arxiv.org/html/<id>v1/x1.png`
    - 通过 scrapling 的 `page.css('img')` 查找 `src` 中包含 `x1.png` 的 img 标签即可定位
    - 这是首选，因为稳定、不依赖外部服务器
2. **项目主页的 teaser / overview 图**
    - 如果论文有项目主页（如 `https://xxx.github.io/`），优先找页面顶部的大图
    - 通过 scrapling 抓取主页，查找首张内容图片（排除 logo、icon）
    - URL 需确认可直接外链访问
3. **GitHub README 中的 demo / teaser 图**
    - 如果论文仓库 README 有代表性截图，可作为备选
    - 注意 GitHub 的图片链接通常是 `https://github.com/.../raw/...` 格式
4. **若以上均无法获取**：省略截图，不强行用无关图片填充

选择标准：
- 优先展示论文整体 pipeline、系统概览或数据集概览的图（而非单张结果特写）
- 图片应能帮助读者一眼判断这篇论文是做什么的
- 避免选择公式、表格、或过于细节的局部截图

#### 论文命名规范

文件名格式：`(venue year) Title.md`

venue 标准化对照：
- SIGGRAPH / SIGGRAPH Asia → `SIGGRAPH` / `SIGGRAPH Asia`
- ACM Transactions on Graphics → `TOG`
- CVPR、ICCV、ECCV、NeurIPS、ICML、ICLR、CHI 等 → 保持不变
- arXiv 预印本（尚未被接收）→ `arXiv`
- 其他会议/期刊 → 使用官方缩写

若论文有多个版本（如 arXiv 预印本后被接收），优先使用最新已知 venue。若暂时只知道 arXiv，用 `arXiv`，后续被接收后再更新文件名。

#### 补全正文中的链接

正文写完后，必须逐项扫描「解决思路」「关键创新」「还原方法」「其他」「核心论文引用」中提到的所有论文、技术、人物名称，逐个判断：

1. 在 `80-archive-档案库/` 中搜索是否已有对应笔记 → 有则用 `[[wikilink]]`
2. 没有笔记但存在明确的链接（论文、代码、主页）→ 用 Markdown 外部链接 `[名称](URL)`
3. 上述两项都没有 → 保留纯文本

此步骤是强制检查项，不可跳过。

## Ambiguity handling

遇到以下情况时，优先保守处理：
- 无法确认是否为同一篇论文：先向用户确认
- 多个来源信息冲突（如 arXiv 版本与会议版本有差异）：优先采用会议/期刊正式版本，标注版本差异
- 论文作者标注有争议（如 equal contribution 标注不统一）：按论文首页原文标注
- 无法确认论文是否已被接收：暂用 `arXiv` 作为 venue，标注"尚未被接收"
- 某栏目暂无可靠信息：可保留空节或不写入该节
- 评分不确定：宁可不评分，也不要随意打分

## Quality bar

- 优先记录"长期有价值"的方法论见解与技术决策，而不是堆链接和摘要
- 核心问题与挑战应说清"为什么要解决这个问题"和"为什么难"
- 解决思路应说清"怎么解决的"和"为什么这么解决有效"，而不是贴方法名称
- 关键创新应明确指出与已有工作的**本质差异**，不只是技术细节的差异
- 还原方法应务实，标注还原成本、可行性和关键限制
- 详细还原步骤应可独立执行，不依赖上下文猜测
- 如果条目很多，先选最能代表此论文的核心内容
- 若发现已有内容结构混乱，先轻量整理，再补充信息
- 输出应便于后续持续补充，而不是一次性塞满

## Delivery

完成后，简要说明：
- 新建了还是更新了哪一篇论文笔记
- 补充了哪些信息，如核心问题、解决思路、关键创新、还原步骤
- 哪些信息仍缺失或待确认（如是否有代码、是否被接收、数据集是否公开等）
- 若已评分，简要说明评分依据
