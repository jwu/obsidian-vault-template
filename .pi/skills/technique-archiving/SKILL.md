# Technique Archiving

为本项目中的技术（语言、框架、库、工具、算法、协议等）建立或完善档案，输出到 `80-archive-档案库/Technique-技术/`，并尽量保持与现有技术笔记一致的结构、语气与信息密度。

## When to use

在以下情况使用本技能：
- 用户要求为某个技术建立档案
- 用户提供一个技术名、GitHub 仓库链接、官网或文档链接，希望整理成标准技术笔记
- 用户要求补充某个技术的原理、同类对比或社区生态
- 用户要求对某项技术按 `1~7` 制打分

## References

开始前优先读取这些参考材料：
- `../../../99-system/templates/Technique Template.md`

如需对齐现有风格，可查看 `80-archive-档案库/Technique-技术/` 下已有笔记，例如：
- `../../../80-archive-档案库/Technique-技术/bevy.md`
- `../../../80-archive-档案库/Technique-技术/mdBook.md`

评分规则参考：
- `../../../99-system/rating.md`（`1~7` 整数制）

## Output

技术档案统一放在：
- `80-archive-档案库/Technique-技术/`

默认要求：
- 文件名使用技术常用名
- 笔记需包含 `created:` 属性
- 优先沿用 `[[Technique Template]]` 的结构
- 语言 wikilink 一律小写，创建前需检查 `80-archive-档案库/Technique-技术/` 中是否已有对应语言笔记

## Rules

- 默认使用简洁中文
- 保留事实，不臆测
- 优先小步补充，不大改原文
- 如果信息不确定，宁可留空，也不要强填
- 编辑 Markdown 内容时保持 Obsidian 兼容
- 引用文件夹时不要使用 `[[wikilink]]`，统一使用 `文件夹名/` 形式
- 若需创建、移动、重命名技术笔记，遵循项目要求，使用 `obsidian vault=work ...` 完成
- 技术评分统一使用 `1~7` 整数制，依据 `99-system/rating.md`
- 笔记内容中提及的技术、人物、概念等，若库内已有对应笔记则使用 wikilink，若是外部资源则使用外部链接，尽量让内容可导航

## Workflow

### 1. 确认输入

用户可能提供：
- 一个技术名（最常见，如「bevy」「ffmpeg」「biome」）
- 技术名 + GitHub 仓库链接
- 技术名 + 官网或文档链接
- 技术名 + 领域描述（如「那个 Rust 写的 linter」）

目标是先确认"要归档的是哪个技术"。

- 如果用户提供的信息已经足够唯一定位到该技术，则继续下一步
- 如果信息过少、存在重名，或无法确认是否为目标技术，则先向用户确认
- 在技术身份未确认前，不要擅自创建笔记或合并到现有笔记

### 2. 确认是否已有笔记

先搜索 `80-archive-档案库/Technique-技术/` 中是否已有同名或近似技术笔记。

- 如果已有：在原笔记上补充
- 如果没有：基于模板新建
- 如果出现同名或疑似重名技术：先暂停并向用户确认，不要擅自合并

### 3. 信息收集与内容整理

#### 基本信息

优先从以下来源获取：
- 官网 / 文档站
- GitHub / GitLab 仓库（README、Wiki、Releases）
- 包管理器页面（npm、crates.io、pypi 等）

获取基础信息时优先使用 `uv run scrapling` 抓取网页内容，GitHub 仓库信息优先使用 `gh` 命令。

#### 原理挖掘

- 从文档的 Architecture / Internals / How it works 章节找
- 从源码 README 或 CONTRIBUTING 中找架构说明
- 如有核心技术（如特有算法、数据结构、编译优化），需单独说明

#### 相关技术

在"其他"栏目中补充：
- **同类技术**：竞争或替代方案，若内部已有档案则用 wikilink
- **依赖/被依赖关系**：该技术依赖了哪些重要技术、被哪些重要项目依赖
- **相关论文 / 文章**：有影响力的论文、博客、技术分析
- **关键事件**：重要的版本发布、架构变更、社区分裂或合并等历史事件
- **作者 / 团队**：核心作者或开发团队简介（若人物不重要可不展开）
- **使用情景与风险**：适合/不适合的场景、已知坑点

### 4. 生产档案文件

#### 建立 frontmatter 属性

优先整理这些属性：

```yaml
---
created: YYYY-MM-DD
template: "[[Technique Template]]"
website:
repository:
desc:
author:
languages:
rating:
license:
aliases:
---
```

填写规则：
- `desc`：一句话说明这个技术是什么、做什么
- `website`：官网或文档站
- `repository`：主仓库链接（GitHub / GitLab 等）
- `languages`：实现语言，以 wikilink 数组形式，如 `["[[rust]]", "[[cpp]]"]`
    - 先在 `80-archive-档案库/Technique-技术/` 中搜索语言笔记，确保 wikilink 指向实际存在的文件名
    - 注意语言可能有多种写法：如 C++ 对应 `cpp`、`c++`、`c plus plus` 等，搜索时需覆盖常见变体
    - 找到对应语言笔记后，直接使用其文件名（不含 `.md`）作为 wikilink，如文件名是 `cpp.md` 则用 `[[cpp]]`
    - 未找到语言笔记时，再用语言名小写作为 wikilink
- `author`：核心作者或维护组织
- `rating`：按 `1~7` 整数制评分；未充分了解时可暂空
- `license`：许可证（如 MIT、Apache-2.0、GPL-3.0）
- `aliases`：仅在确有常见别名、缩写、旧名时填写
- 未找到或无法确认的字段，直接省略，不要保留空字段
- 仅保留已确认且对后续检索、补充有价值的属性

#### 使用统一格式写入正文

```md
## 简介

介绍一下这个技术。

## 特点

- 技术特点
- 基于什么语言、技术框架开发
- 核心技术是什么
- 有哪些功能
- 有什么优势和劣势

## 原理

- 简单介绍实现原理

## 生态

> 语言类技术建议补充此节，非语言类技术可省略。参考已有语言笔记（如 [[rust]]、[[cpp]]）的生态栏格式。

### 上手

- 工具链管理、包管理、安装方式等

### 编辑

- LSP、Tree-sitter、格式化工具、Lint 等

### 编译

- 编译器、交叉编译支持等

### 运行

- 如何运行/部署

### 测试

- 单元测试框架、基准测试工具等

### 调试

- 调试器、性能分析工具等

## 其他

- 同类技术（内部已有的用 wikilink）
- 相关论文、文章、知识笔记
- 关键设计决策、历史事件、社群探讨
- 使用情景、风险点、优势与劣势的补充
```

格式要求：
- 简介：简要介绍这个技术
- 特点：用无序列表，每个特点一行；涵盖技术特点、技术栈、核心技术、功能、优劣势
- 原理：简要说明实现原理
- 生态：语言类技术按「上手 → 编辑 → 编译 → 运行 → 测试 → 调试」六节分项列出，每节列出具体工具/方案，内部已有的用 wikilink，外部的用 Markdown 链接；非语言类可省略此节
- 其他：使用无序列表，注意不要重复属性中已有的信息（如仓库链接、开发语言）
- 如某栏目暂无可靠信息，可保留空节或不写入该节

#### 补全正文中的链接

正文写完后，必须逐项扫描「特点」「原理」「生态」「其他」中提到的所有技术、框架、库、工具、协议、语言、概念名称，逐个判断：

1. 在 `80-archive-档案库/` 中搜索是否已有对应笔记 → 有则用 `[[wikilink]]`
2. 没有笔记但存在明确的官网或仓库 → 用 Markdown 外部链接 `[名称](URL)`
3. 上述两项都没有 → 保留纯文本

扫描范围包括但不限于：
- 底层依赖（如解析器、模板引擎、搜索引擎库）
- 同类技术（如「类似 XX」「与 XX 对比」）
- 技术标准或协议（如 CommonMark、WebAssembly）
- 编程语言（如 Rust、Python）
- 相关工具链（如特定的 CLI 工具、构建系统）

此步骤是强制检查项，不可跳过。

## Ambiguity handling

遇到以下情况时，优先保守处理：
- 无法确认是否为同一技术：先向用户确认
- 多个来源信息冲突：优先采用官网或仓库 README；若仍无法判断，则暂不写入
- 技术版本差异大：优先记录当前稳定版本的信息，必要时标注版本
- 某栏目暂无可靠信息：可保留空节或不写入该节
- 评分不确定：宁可不评分，也不要随意打分

## Quality bar

- 优先记录"长期有价值"的技术特征与设计决策，而不是堆链接
- 简介应让读者快速判断"这个技术是否适合我的场景"
- 原理应说明"怎么做到的"，而不只是"做了什么"
- 其他中的同类对比应有实际差异点，而不是泛泛的"类似 XX"
- 如果条目很多，先选最能代表此技术的少量信息
- 若发现已有内容结构混乱，先轻量整理，再补充信息
- 输出应便于后续持续补充，而不是一次性塞满

## Delivery

完成后，简要说明：
- 新建了还是更新了哪一篇技术笔记
- 补充了哪些信息，如官网、仓库、原理、同类对比
- 哪些信息仍缺失或待确认
- 若已评分，简要说明评分依据
